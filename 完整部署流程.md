# 🚀 完整部署流程 - 从代码提交到域名访问

本文档提供从代码提交到Cloudflare Pages部署和域名配置的完整步骤。

## 📋 前置条件检查

在开始之前，请确认：

- [x] ✅ Hugo已安装并可以正常运行
- [x] ✅ 博客代码已准备就绪
- [x] ✅ 拥有GitHub/GitLab账户
- [x] ✅ 拥有Cloudflare账户
- [x] ✅ 拥有域名（已配置到Cloudflare DNS）
- [x] ✅ `config.yaml` 中的 `baseURL` 已配置为你的域名

**当前配置的域名：** `mingxinwang.com`

## 第一部分：Git代码提交

### 步骤1：检查Git状态

```bash
# 进入项目目录
cd /Users/moneywang/blog

# 检查Git状态
git status
```

### 步骤2：创建GitHub仓库

1. 访问 [GitHub](https://github.com)
2. 点击右上角 **"+"** → **"New repository"**
3. 填写仓库信息：
   - **Repository name**: `my-blog` (或你喜欢的名字)
   - **Description**: `Personal blog built with Hugo`
   - **Visibility**: Public 或 Private（推荐Public，免费）
   - **不要**勾选 "Initialize this repository with a README"（我们已经有了）
4. 点击 **"Create repository"**

### 步骤3：提交代码到本地仓库

```bash
# 1. 添加所有文件（.gitignore会自动排除public等）
git add .

# 2. 检查将要提交的文件
git status

# 3. 提交代码
git commit -m "Initial commit: Hugo blog with PaperMod theme"

# 4. 确认提交成功
git log --oneline
```

### 步骤4：推送到GitHub

```bash
# 1. 添加远程仓库（替换为你的GitHub用户名和仓库名）
git remote add origin https://github.com/yourusername/my-blog.git

# 2. 设置主分支为main
git branch -M main

# 3. 推送到远程仓库
git push -u origin main
```

**如果遇到认证问题：**
- 使用Personal Access Token（推荐）
- 或使用SSH密钥：`git@github.com:yourusername/my-blog.git`

### 步骤5：验证推送成功

1. 访问你的GitHub仓库页面
2. 确认所有文件都已上传
3. 确认 `themes/PaperMod` 显示为submodule（如果有）

---

## 第二部分：Cloudflare Pages部署

### 步骤1：登录Cloudflare Dashboard

1. 访问 [Cloudflare Dashboard](https://dash.cloudflare.com/)
2. 使用你的Cloudflare账户登录

### 步骤2：创建Pages项目

1. 在左侧菜单中找到 **"Workers & Pages"**
2. 点击 **"Pages"**
3. 点击 **"Create a project"**
4. 选择 **"Connect to Git"**

### 步骤3：连接Git仓库

1. **选择Git提供商**
   - 选择 GitHub 或 GitLab
   - 点击 **"Authorize Cloudflare"** 授权

2. **选择仓库**
   - 在仓库列表中找到你的博客仓库（如：`my-blog`）
   - 点击 **"Begin setup"**

### 步骤4：配置构建设置

在 **"Set up builds and deployments"** 页面配置：

#### 基本设置

- **Project name**: `my-blog` (或自定义名称)
- **Production branch**: `main`

#### 构建设置

- **Framework preset**: 选择 `Hugo`
- **Build command**: `hugo --minify`
- **Build output directory**: `public`
- **Root directory**: `/` (留空)

#### 环境变量（可选但推荐）

点击 **"Add environment variable"** 添加：

- **Variable name**: `HUGO_VERSION`
- **Value**: `0.147.4` (或你安装的Hugo版本，运行 `hugo version` 查看)

**为什么需要HUGO_VERSION？**
- 确保构建环境使用正确的Hugo版本
- 避免版本不匹配导致的构建错误

### 步骤5：保存并部署

1. 点击 **"Save and Deploy"**
2. 等待构建完成（通常1-3分钟）
3. 构建成功后，你会看到一个预览URL，格式如：`https://my-blog.pages.dev`

### 步骤6：验证构建成功

1. 点击构建日志查看详细信息
2. 访问预览URL确认网站正常显示
3. 检查是否有错误或警告

**常见构建问题：**
- ❌ 主题找不到 → 检查submodule是否正确配置
- ❌ 构建失败 → 查看构建日志中的错误信息
- ⚠️ 警告 → 通常可以忽略，但建议查看

## 第三部分：配置自定义域名

### 步骤1：准备域名

确认你的域名已在Cloudflare DNS中：
1. 在Cloudflare Dashboard中选择你的域名
2. 进入 **"DNS"** 部分
3. 确认域名记录已配置

### 步骤2：在Pages中配置域名

1. 在Cloudflare Pages项目页面
2. 点击 **"Custom domains"** 标签
3. 点击 **"Set up a custom domain"**
4. 输入你的域名：
   - 根域名：`mingxinwang.com`
   - 或子域名：`blog.mingxinwang.com`（推荐）

### 步骤3：DNS自动配置

Cloudflare会自动：
1. 创建CNAME记录指向Pages
2. 配置SSL证书（自动HTTPS）
3. 设置DNS代理（橙色云朵）

**等待时间：**
- DNS配置：通常1-5分钟
- SSL证书：通常5-15分钟（最多24小时）

### 步骤4：验证域名配置

1. **检查DNS记录**
   - 在Cloudflare DNS中查看是否自动创建了CNAME记录
   - 记录应该指向 `your-project.pages.dev`

2. **检查SSL证书**
   - 在Pages项目的 **"Custom domains"** 中查看SSL状态
   - 应该显示 "Active certificate"

3. **测试访问**
   - 访问你的域名：`https://mingxinwang.com`
   - 确认网站正常显示
   - 确认HTTPS正常工作（浏览器显示锁图标）

### 步骤5：更新config.yaml中的baseURL

如果域名配置成功，确认 `config.yaml` 中的 `baseURL` 正确：

```yaml
baseURL: 'https://mingxinwang.com'  # 确保与你的域名一致
```

如果之前配置不正确，需要更新：

```bash
# 1. 编辑config.yaml，更新baseURL
# 2. 提交更改
git add config.yaml
git commit -m "Update baseURL to production domain"
git push
```

Cloudflare Pages会自动检测到代码更新并重新部署。

## 第五部分：自动化部署

### 自动部署机制

配置完成后，每次推送代码到 `main` 分支，Cloudflare Pages会自动：

1. **检测代码变更**
   - 监听GitHub webhook
   - 检测到push事件

2. **运行构建**
   - 拉取最新代码
   - 运行 `hugo --minify`
   - 生成静态文件

3. **部署更新**
   - 部署到全球CDN
   - 更新网站内容

4. **通知结果**
   - 发送构建状态通知（如果配置了）
   - 在Dashboard中显示构建历史

### 工作流程示例

```bash
# 1. 创建新文章
hugo new posts/my-new-article.md

# 2. 编辑文章内容
# ... 编辑文件 ...

# 3. 本地预览（可选）
hugo server -D

# 4. 提交并推送
git add .
git commit -m "Add new article: my-new-article"
git push

# 5. 等待自动部署（1-3分钟）
# Cloudflare Pages会自动检测并部署
```

## 第六部分：常见问题解决

### 问题1：构建失败 - 找不到主题

**错误信息：**
```
Error: Unable to locate config file or config directory
```

**解决方案：**

1. **检查submodule配置**
   ```bash
   # 确认.gitmodules文件存在
   cat .gitmodules
   
   # 确认主题目录存在
   ls themes/PaperMod
   ```

2. **在Cloudflare Pages中启用submodule**
   - 在项目设置中找到 **"Builds"**
   - 在环境变量中添加：
     - **Variable name**: `NPM_FLAGS`
     - **Value**: `--legacy-peer-deps`
   - 或者在构建命令前添加：
     ```bash
     git submodule update --init --recursive && hugo --minify
     ```

3. **或者直接提交主题文件**
   ```bash
   # 删除submodule，直接提交主题
   git submodule deinit themes/PaperMod
   git rm --cached themes/PaperMod
   git add themes/PaperMod
   git commit -m "Include theme directly instead of submodule"
   git push
   ```

### 问题2：样式丢失

**症状：** 网站可以访问，但没有样式

**解决方案：**

1. **检查config.yaml**
   ```yaml
   theme: PaperMod  # 确保主题名称正确
   ```

2. **检查baseURL**
   ```yaml
   baseURL: 'https://mingxinwang.com'  # 确保是HTTPS
   ```

3. **清除缓存并重新构建**
   - 在Cloudflare Pages中触发重新部署
   - 或推送一个空提交：
     ```bash
     git commit --allow-empty -m "Trigger rebuild"
     git push
     ```

### 问题3：域名无法访问

**症状：** DNS已配置，但无法访问

**解决方案：**

1. **检查DNS记录**
   - 在Cloudflare DNS中确认CNAME记录存在
   - 记录应该指向 `your-project.pages.dev`

2. **检查SSL证书**
   - 在Pages项目的Custom domains中查看SSL状态
   - 如果显示"Pending"，等待最多24小时

3. **检查域名代理状态**
   - DNS记录应该是橙色云朵（代理开启）
   - 如果是灰色，点击云朵图标开启代理

4. **清除DNS缓存**
   ```bash
   # macOS
   sudo dscacheutil -flushcache
   sudo killall -HUP mDNSResponder
   
   # 或使用在线工具检查DNS传播
   # https://www.whatsmydns.net/
   ```

### 问题4：构建时间过长

**解决方案：**

1. **优化构建命令**
   - 使用 `hugo --minify` 而不是 `hugo`
   - 避免构建草稿：不要使用 `-D` 参数

2. **检查文件大小**
   - 确保没有提交大文件
   - 使用 `.gitignore` 排除不必要的文件

3. **使用缓存**
   - Cloudflare Pages会自动缓存依赖
   - 首次构建较慢，后续会更快

### 问题5：自动部署不工作

**解决方案：**

1. **检查GitHub webhook**
   - 在GitHub仓库设置中查看Webhooks
   - 确认Cloudflare的webhook存在且正常

2. **手动触发部署**
   - 在Cloudflare Pages中点击 **"Retry deployment"**
   - 或推送一个空提交触发

3. **检查分支设置**
   - 确认Production branch设置为 `main`
   - 确认推送的是正确的分支
